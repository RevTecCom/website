How to License a Work as Common Software
_________________________________________________________________________



Licensing a work under Common Software is only one aspect of the contribution of the technical intelligentsia to the proletarian movement, but it is an important aspect, legally speaking, because it grants certain rights and introduces certain limitations to the users of the work under bourgeois law, acting as a sort of wedge, and scaring a few lawyers away.

The steps, in a summarized way, are the following:

  • Get a non-interest document or/and using a pseudonym

  • Include copyright and license notices in both the program and the source-code

  • Add a LICENSE file with a copy of the CPL


Non-Interest Document and Pseudonym

When possible, release the work under a pseudonym to stop it from being linked to a real identity, and remove any aspect from it that may imply a real identity in its creation. This is so a creator of the work can avoid an employer, school or anyone associated with themselves from falsely claiming rights over the software via means of "proving" their association to them. Pseudonyms are protected under international copyright law.

If the work is non-trivial and was used as part of a paper, teaching assignment, teaching example, or was previously incorporated as part of a product, a creator may need to acquire a signed document where the company or institution disavows their interest in the work. The document does not need to specify what the work is intended to be used for or which license it will be released under, keep it as vague as possible. Here is an example:

The Institute of Digging Very Deep Holes hereby disclaims all copyright interest in the program “Boredig”, written by Adam Smith.

signature of Harry Hol, 1 April 2005
Harry Hol, Director of the Institute of Digging Very Deep Holes

Copyright and License Notices

Programs should contain copyright notices and license statements which make it clear that the program is released under a Common Software license. The copyright years should be the years in which a version of the program was prepared for release, not the year of publishing. Include every year where a release occurred in the copyright notice (so, if a program has had a release every three years from 2013, the years listed would be "2013, 2016, 2019"). No matter your language, it is preferrable to use the word "copyright", as it is an internationally recognized legal term.

In a compiled program, it is of the discretion of the legal holder of the work to decide where to include the copyright and license notice, but it is recommended you include it at least once if reasonable. If you have to choose between including the copyright and license notice, include the copyright notice.
For command line interface programs, it is common to show both the copyright and license notices if the program is invoked with the "--version" or "-v" argument, and then terminate the program. For common UI programs, create an "info" or "help" button or tab that opens a window containing the notices. For graphical programs like video-games, show the disclaimers in small text on a startup screen.

In source-code form, the copyright and license notices should be included at the beginning of every source file in the form of a comment (though an abridged version can be used for everything but the "main file", if applicable). Here is an example of both the copyright and license notices:

Copyright (c) 2005 Adam Smith. All rights reserved.

Boredig is common software: you can redistribute it and/or modify it under the terms of the Commons Protection License as published by the Revolutionary Technical Committee.

This program is distributed in the hope that it will be useful, but WITHOUT ANY WARRANTY; without even the implied warranty of MERCHANTABILITY or FITNESS FOR A PARTICULAR PURPOSE. See the Commons Protection License for more details.


Adding a LICENSE file

A copy of the license along with the program. For the majority of distributions, this is as easy as including a file named "LICENSE", containing a plaintext version of the license itself. In case the program is distributed in a very limited physical medium where it is unreasonable to include a copy of the license inside, the license can be printed and distributed along with the medium in paper format, or a small card can be used instead, containing a URL to a copy of the license. The "LICENSE" file must be included in both binary and source-code form distributions.
